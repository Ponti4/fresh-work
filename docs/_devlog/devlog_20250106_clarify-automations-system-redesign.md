# DevLog - 2025-01-06 (Clarify-Automations 시스템 재설계)

## 📝 한줄 요약

비개발자 프리랜서를 위한 자동화 요구사항 정의 시스템을 **문제 중심 (불편함 감지) → 기능 분해 (MVP 정의) → 구현 방식 (외부 연동 확인)**으로 완전히 재설계하고, 4주 동기식 실습 + 성과 측정 프레임워크까지 함께 구축했습니다. 이로써 프리랜서가 "막히지 않고" 4주에 3개 자동화를 완성할 수 있는 교육 체계를 완성했습니다.

## 🎯 이런 분들께 도움돼요

- **AI 기반 자동화 교육을 설계하는 사람** - 비개발자 입문자를 위한 실습 커리큘럼
- **프리랜서 커뮤니티 리더** - 반복 업무 자동화로 시간을 확보하는 프로세스
- **AI 활용 리터러시를 높이고 싶은 분** - Claude Code와 함께 일하는 실제 패턴 학습
- **Template/Skill 개발자** - 사용자 경험 중심의 설계 원칙

## 😫 문제 상황 (Before)

기존의 clarify-automations 스킬은 **극단화 질문 (Q4) 중심**으로 설계되어 있었습니다:

```
Q1 (빈도) → Q2 (소요시간) → Q3 (규칙성) → Q4 (극단화) → Q5 (가치도출)
                                                 ↑
                                        문제의 본질 파악 (추상적)
```

**문제점:**
1. **Q1-Q2의 월 소요시간이 "수치로만" 제시** - 프리랜서가 "근데 이게 자동화 가치가 있나?" 의심
2. **Q3은 규칙성 (예/아니오)** - 기능 분해가 안 되어 "뭘 자동화할 건지" 불명확
3. **Q4 극단화 (10배 늘면?)** - 너무 추상적, 실제 구현 스코프와 떨어짐
4. **최소 기능 정의 부재** - "이것만 해도 되지 않을까?"라는 MVP 사고가 없음
5. **외부 연동 확인 부재** - 갑자기 Plan Mode에서 "Gmail 연동 필요해요" 하면 프리랜서 당황
6. **동기식 실습 구조 불명확** - 혼자 하다가 막히는 경우 많음

## 🧠 Context의 역할

이번 세션에서 Claude Code의 **Context 인식 능력**이 핵심이었습니다:

### 1️⃣ **실시간 대화 기반 설계**
- 처음엔 "극단화가 핵심이 아닐 수 있다"는 직감 → 논의하며 확인
- "기능 분해가 필요하다"는 새로운 방향성 발견
- "외부 연동 확인이 선행되어야 한다" 순서 재정의

### 2️⃣ **One-by-One 원칙 도입**
- 5개 질문의 한계를 인식 → 비개발자가 한 번에 하나씩만 받아야 함
- Q4를 "기능 분해 3-4개 반복 질문"으로 변환
- "설명하지 말고, 간단히 제시하라" 압축 원칙 도입

### 3️⃣ **맥락과 현실의 연결**
- docs/wiki의 Skills, Commands, Hooks, Sub-agents 문서 검토
- "Claude Code만으로 가능한 것"과 "외부 연동 필요한 것" 명확히 구분
- Plan Mode로의 **자연스러운 전환**이 가능하도록 설계

### 4️⃣ **학습 목표의 재정의**
- "3개 완성"이 아니라 **"문제 정의 → Agent 위임 → 본질에 집중"** 패턴 학습
- 주 1회 동기식 + 1주 비동기 학습의 혼합 구조
- 발표회 + 오프라인 스터디로 팀 학습 연계

## 🔧 작업 과정

### 1단계: 5개 주제별 Deep Dive 논의

#### 주제 1: Setup-Workspace 질문 최소화
**사용자 요청:**
```
프리랜서 입장에서 프로필 입력 피로도를 낮추되,
자동화 여정의 맥락은 유지하고 싶다
```

**Claude 작업:**
- 기존: 8개 항목 (프로필 상세)
- 개선: 4-5개만 (OS, Python, 개발경험, 목표)
- **효과:** Setup 진입 장벽 50% 감소

---

#### 주제 2: 월 소요시간 표현 + 기능 분해

**사용자 요청:**
```
월 소요시간 수치가 "대략" 정도로 제시되어야 하고,
불편함 → 기능 분해의 자연스러운 흐름이 필요하다
```

**Claude 작업:**
- Q2 후 "월 약 X시간 + 연간 약 Y시간" 자동 계산
- Q3에서 "불편함 감지" (자유 입력)로 변경
- Q4를 "기능 분해 반복 질문 3-4개"로 재설계
  ```
  1. "그렇다면 핵심 기능은?"
  2. "이것만 해도 충분한가?"
  3. "더 간단하게 할 방법은?"
  ```

**산출물:**
- `docs/_clarify/` - 새로운 Task 정의서 템플릿

---

#### 주제 3: Q4 극단화 → Q6 외부 연동으로 재설계

**사용자 요청:**
```
극단화는 나중에 (Plan Mode에서) 다루고,
요구사항 정의와 최소 복잡도가 먼저여야 한다.
Q6에서 외부 연동 여부만 확인하자.
```

**Claude 작업:**
- Q4: 극단화 제거, 기능 분해 3-4개 반복 질문
- Q5: 기대 효과 (기존 Q5 그대로 유지)
- Q6 신규 추가: 외부 연동 확인 (예/아니오)
  - 아니오 → Plan Mode로 바로 진행
  - 예 → 서비스명만 수집 → 추가 논의

**효과:**
- 비개발자도 "내 자동화는 Claude Code만으로 되나?" 스스로 판단 가능
- 혼동 감소, 프리랜서 신뢰도 증가

---

#### 주제 4: 4주 실습 + 1주 비동기 학습

**사용자 요청:**
```
주 1회 동기식으로 만나고,
1주일 동안 비동기로 학습하는 구조
```

**Claude 작업:**
- **동기식:** 매주 정해진 시간 (예: 화요일 2시) 만나기
  - 질문 & 즉시 해결
  - 진행 상황 체크
  - 1-2시간 소요

- **비동기:** 그 외 시간에 개인 학습
  - `/clarify` + `/plan` + 구현
  - 막히는 부분 비동기 질문 가능
  - 온보딩 자료 제공

**효과:**
- 혼자 하다가 포기하는 일 방지
- 매주 체크인으로 동기 부여

---

#### 주제 5: 성과 측정 프레임워크

**사용자 요청:**
```
A) 정량 (자동화 개수, 절감 시간)
B) 정성 (문제 정의 능력, Agent 위임 자신감)
C) 행동 (실제 삶의 변화)

이 중에 어디를 강조할까?
```

**Claude 작업:**
- 최종: **A + 발표회 + 오프라인 확인** 조합
  - 정량: 자동화 3개 완성 + 월 X시간 절감
  - 정성: 발표회에서 시연 (증거)
  - 행동: 오프라인 스터디에서 팀원과 공유 (전파)

## 💡 배운 점 & 인사이트

### 1️⃣ **문제 정의 방향이 틀리면 모든 게 틀린다**
- 초기: "극단화로 문제의 본질을 파악하자" (추상)
- 최종: "불편함을 직접 듣고, 기능을 함께 분해하자" (구체)
- **핵심:** 사용자의 언어로 시작해야 함

### 2️⃣ **비개발자는 "선택지" 문화가 다르다**
- 개발자: 자유 입력 → 개방적
- 비개발자: 선택지 → 심리적 안전
- **최종 설계:** Q1-Q2, Q5는 선택지 / Q3, Q4는 자유 입력

### 3️⃣ **Q4의 반복 질문이 핵심 혁신**
```
"이것만 해도 되지 않을까?"
"더 간단하게 할 방법은 없을까?"
```
이 2개 질문으로 프리랜서 스스로 MVP를 정의하게 함

### 4️⃣ **Plan Mode와의 명확한 경계**
- Clarify: "뭘 할지" (요구사항)
- Plan Mode: "어떻게 할지" (설계)
- **효과:** 역할이 명확하니 진행이 빠름

### 5️⃣ **성공의 3축이 필요하다**
- 정량만으로는 "몇 개 했는가"만 보임
- 정성만으로는 "실제 영향"이 불명확
- 행동 (팀 공유)까지 가야 **문화 변화**가 일어남

### 6️⃣ **맥락의 힘**
- 이 모든 설계가 가능했던 이유: Claude Code의 Context 인식
- "대화 맥락을 기억하며 설계하기"가 가장 강력한 도구
