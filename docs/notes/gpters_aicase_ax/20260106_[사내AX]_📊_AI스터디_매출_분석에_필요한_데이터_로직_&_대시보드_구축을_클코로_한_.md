# [사내AX] 📊 AI스터디 매출 분석에 필요한 데이터 로직 & 대시보드 구축을 클코로 한 방에!

- **URL**: https://www.gpters.org/dev/post/build-data-logic-dashboard-z2kmYZ8zkCoYtLQ

비개발자가 Claude Code로 오전 2시간 만에 매출 대시보드를 만들었다

## 📝 한줄 요약

Airtable 결제 데이터를 연동해서 기수별 매출/신규율을 한눈에 보는 대시보드를 만들고, 3,590개 레코드의 신규/재참여 분류까지 자동화했다.

## 🎯 이런 분들께 도움돼요

- Airtable, 노션, 구글시트에 데이터는 쌓이는데 활용을 못 하고 있는 분

- 매번 스프레드시트로 현황 정리하는 게 지겨운 분

- "개발자 없이도 될까?" 고민 중인 비개발자



## 😫 문제 상황

나는 개발자가 아니다. 그런데 Airtable에 쌓인 결제 데이터를 볼 때마다 답답했다.

기수별 매출이 얼마인지, 신규 고객이 몇 명인지 보려면 매번 필터를 바꿔가며 확인해야 했다. 스프레드시트로 따로 정리하면 되긴 하는데, 그게 또 일이었다.

신규/재참여 구분도 사람이 일일이 판단해서 Airtable에 입력하고 있었다. 기수가 쌓일수록 과거 데이터 정합성도 엉망이 됐다.

## 💡 시작 계기

20기 모집이 다가오면서 더 이상 미룰 수 없었다. 현황을 빠르게 파악해야 하는데, 매번 수동으로 하기엔 시간이 없었다.

그래서 Claude Code에게 던져봤다.

## 🔧 작업 과정

### "에어테이블 연동해줘" - 이게 전부였다

회사에서 Claude Code Max 플랜을 지원해줘서 쓰고 있었는데, 일단 해보자는 마음으로 한 마디 던졌다.

에어테이블을 연동해줘.

솔직히 몇 번의 대화가 오갈 줄 알았다. "어떤 테이블이요?", "API 키는 있나요?" 같은 질문이 올 줄 알았는데, Claude는 그냥 바로 만들기 시작했다.

물론 .env 파일은 직접 만들어서 값을 넣어줘야 했다. Airtable API Key랑 Base ID 두 개.

SDK 설치하고, API 라우트 만들고, 데이터 불러오는 코드까지. 몇 분 만에 끝났다.

'이게 된다고?'


### 0원 결제자 분리하기 - 멤버십 구조 설명

대시보드 뼈대가 생겼는데, 통계가 이상했다. 0원으로 결제된 사람도 매출에 포함되고 있었다.

근데 0원으로 결제된 사람은 분리해서 봐야할 것 같아. 0원 결제로 등록한 멤버 중 참여기수 필드에 membership이라는 단어가 있으면 멤버십 구매자라서 다음 기수는 무료등록 되는 구조거든...

우리 비즈니스 구조를 설명해주니까 Claude가 바로 이해했다. 일반 결제자, 멤버십 무료등록, 기타 무료 세 가지로 구분해서 통계 카드를 다시 구성해줬다.


### 신규/재참여 자동 분류 - 예시 없이 설명하면 못 알아듣는다

신규/재참여/직전기수 구분을 자동화하고 싶었다. 지금까지 이걸 사람이 일일이 판단해서 Airtable에 입력하고 있었는데, 너무 비효율적이었다.

신규나 재참여냐 직전기수참여자냐가 중요해. 멤버 참여기수 필드를 보고 판단할 수 있을 거 같은데 기획해줘.

처음에 Claude한테 설명했는데 로직을 잘못 이해했다. 내가 설명을 대충 했던 것 같다.

다시 이렇게 말했다.

만약 15기 결제데이터라면 실제참여기수 필드가 15, 16, 17이 있다면 15가 신규고.. 14가 있으면 직전기수고 알지?

구체적인 예시를 들어주니까 바로 이해했다.

도메인 지식이 필요한 로직은 "이런 경우엔 이거야"라고 예시를 들어줘야 한다. 그냥 개념만 설명하면 잘못 구현할 확률이 높다.

![파란색 배경의 한국어 페이지 스크린샷](https://tribe-s3-production.imgix.net/TMQ7Y75XDf42hoUIaxmZV?auto=compress,format)


### Airtable 자동 업데이트 - 일부만 되는 문제

계산한 결과를 Airtable 필드에도 업데이트해달라고 했다.

신규/재참여 라는 필드가 있어. 너가 멤버 유형별 count 한거를 해당 필드에 업데이트 해줄 수 있어?

근데 일부만 업데이트되고 전체가 안 됐다.

근데 일부만 되고 다 안채워지는데 이유가 뭐야? 개선해

Claude가 원인을 찾았다. Airtable API가 초당 5회 제한이 있어서 Rate Limit에 걸린 거였다. 배치 간 딜레이를 추가해서 해결했다.


### 마케터의 눈을 빌리다 - UX 개선

대시보드에 결제 명단이 쭉 나열되어 있어서 스크롤을 한참 내려야 했고, 어떤 숫자를 봐야 하는지도 잘 몰랐다.

나는 마케터가 아니다. 어떤 지표가 중요한지 감이 없었다.

ultrathink 대시보드에서 결제명단을 다 보여주니까 스크롤을 너무 내려야 해. 너가 마케터라고 생각하고 어떤 지표를 봐야하는지 보고 대시보드 기획을 해줘.

"ultrathink"는 Claude가 더 깊이 생각하게 만드는 키워드다. 붙이니까 확실히 답변이 달랐다.

Claude는 마케터 관점에서 봐야 할 핵심 지표를 정리해줬고, 상단에 KPI 바를 만들고, 멤버 구성을 도넛 차트로 시각화하고, 긴 테이블은 페이지네이션으로 정리해줬다.

내가 몰랐던 관점을 빌려올 수 있다는 게 신기했다.

### 기수별 추이 차트 - "그려줘"하면 된다

스프레드시트에서 차트 그리는 게 얼마나 귀찮은지 아는 사람은 안다. 축 설정, 범례 위치, 색상 맞추기... 매번 시간을 잡아먹었다.

기수별 추이 차트 추가해줘.

끝이었다. Recharts라는 라이브러리로 등록자 추이, 멤버 유형별 추이, 사전판매 추이까지 깔끔하게 그려줬다.

### 사전판매 기간 분석 - 테이블 간 연결

사전판매 기간 동안의 유효 결제를 따로 보고 싶었다.

에어테이블 '기수관리' 테이블 보고 사전판매시작일 ~ 모집시작일 날짜기준 0원결제가 아닌 것의 갯수를 파악할 수 있니?

Claude가 기수관리 테이블에서 날짜를 가져와서 결제 테이블과 연결했다. 사전판매 기간 유효결제 39건, 여기서도 신규/직전기수/재참여 비율까지 대시보드에 추가해줬다.


### 로딩 속도 개선 - 내 생각을 말해도 된다

처음 접속할 때 로딩이 너무 길었다. 전체 기수 데이터를 다 불러오고 있었기 때문이다.

처음 접속시 로딩이 너무 길거든? 내 생각엔 우선 20기에 해당하는 것만 먼저 로딩하고, 기수드롭다운 변경하면 그때 다른 것도 하나씩 로딩하자

내가 해결책까지 제안했더니 Claude가 바로 구현해줬다. 기수별 Lazy Loading으로 초기 로딩 속도가 확 빨라졌다.


### Vercel 배포 - "너가 직접 봐줘"가 통한다

Vercel에 배포했는데 404 에러가 났다. 고쳤다. 또 났다. 또 고쳤다. 또 났다.

짜증이 났다.

너가 버셀도 직접 봐주면 안돼?

반쯤 짜증 섞인 말이었는데, Claude가 진짜로 Vercel CLI에 로그인해서 확인하기 시작했다. 그리고 원인을 찾아냈다. Framework Preset이 제대로 인식 안 되고 있었고, vercel.json 파일이 커밋에서 누락되어 있었다.

에러 메시지만 보여주는 것보다 "직접 확인해달라"고 하는 게 더 빠를 때가 있다.


### 같은 실수를 반복하면 가이드라인을 만든다

근데 vercel.json 누락을 또 했다. 배포할 때마다 같은 실수를 반복하고 있었다.

이거 왜 자꾸 반복해서 실수해? 다시 실수 안 하게
[CLAUDE.md](http://CLAUDE.md)
에 추가해줘.

Claude가 재발 방지 가이드라인을 만들어서
[CLAUDE.md](http://CLAUDE.md)
파일에 정리해줬다. "git status로 untracked 파일 확인하기", "vercel.json 커밋 누락 주의" 같은 체크리스트였다.

이후로 같은 실수는 안 났다.

AI도 실수한다. 근데 그 실수를 문서화해두면 다음엔 안 한다.


### 전체 기수 데이터 정리 - 수작업으로는 며칠 걸렸을 일

사실 오래된 데이터들은 수동으로 수정한 것들이 많아서 정합성이 안 맞는 경우가 많았다. 어떤 기수는 신규/재참여 구분이 되어 있고, 어떤 기수는 안 되어 있고.

그러면 모든 기수에 대해서 해당 api 실행해줘

코드로 로직을 짜니까 전체 기수에 대해 일괄 재계산할 수 있었다. ( 몇 천 건 )

수작업으로 했으면 며칠 걸렸을 일이다.

### 팀 협업의 변화

대시보드가 생기고 나서 협업 방식이 달라졌다.

같이 일하는 마케터가 "이 지표도 보고 싶어"라고 하면 바로 추가해줄 수 있었다. 예전 같았으면 스프레드시트 열어서 한참 작업했을 텐데, 이제는 Claude한테 말 한 마디면 된다.

그리고 팀원들이 같은 대시보드를 보면서 이야기할 수 있게 됐다. 각자 스프레드시트로 정리하던 시절엔 "내 데이터랑 네 데이터가 왜 달라?" 같은 일이 있었는데, 이제는 한 곳만 보면 된다.


## 💬 이 과정에서 배운 AI 활용 팁

### 효과적이었던 것

- 도메인 로직은 예시와 함께 설명해야 한다
- "15기 결제인데 참여기수가 14가 있으면 직전기수야" 처럼

- 반복되는 실수는
[CLAUDE.md](http://CLAUDE.md)
에 가이드라인으로 문서화
- AI도 같은 실수를 반복한다. 문서화하면 안 한다

- 에러가 반복되면 "직접 봐달라"고 요청
- 에러 메시지 복붙보다 빠를 때가 있다

- 모르는 관점은 역할을 맡겨서 빌려온다
- "너가 마케터라고 생각하고..."

- 내 아이디어를 먼저 제안해도 된다
- "내 생각엔 이렇게 하면 어떨까" 하면 바로 구현해준다


### 이렇게 하면 안 돼요

- 로직을 모호하게 설명하면 잘못 구현한다
- 구체적인 예시 필수

- "안 돼"라고만 하면 원인을 못 찾는다
- 에러 메시지나 스크린샷을 보여줘야 한다
