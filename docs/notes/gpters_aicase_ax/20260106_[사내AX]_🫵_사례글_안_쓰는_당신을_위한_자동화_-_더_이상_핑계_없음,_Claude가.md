## 🔧 작업 과정

### 전체 흐름 요약

- 세션 파일 발견
→ Claude Code 대화 내역이 프로젝트별로 저장된다는 걸 알게 됨

- /devlog 커맨드 생성
→ 세션 파일을 파싱해서 개발 로그 자동 생성

- /ai-case 커맨드 생성
→ DEVLOG 기반으로 사례글 자동 작성 워크플로우 설계

- 스토리텔링 포맷 개선
→ 딱딱한 구조 대신 자연스러운 서술체로 변경

- 맥락 누락 문제 해결
→ 사용자 질문을 맥락과 함께 기록하도록 규칙 추가



### 상세 작업 과정

### 1. 세션 파일 발견 - "대화 내역이 저장된다고?"

대시보드 사례글을 쓰려고 했는데, 대화 내역을 다시 찾는 게 너무 힘들었어요. Claude한테 물어봤죠.

클로드 대화내역 저장하는 곳 있지 않아? 프로젝트별로

Claude가
`~/.claude/projects/{프로젝트경로}/*.jsonl`
경로를 알려줬어요. 프로젝트별로 세션 파일이 저장된다는 걸 그때 처음 알았습니다.

"이걸 파싱하면 자동으로 개발 로그를 만들 수 있겠다!" 싶었어요. 대화 내역 찾느라 시간 낭비할 필요가 없어지는 거죠.

![](https://tribe-s3-production.imgix.net/wqkhvWo4O3Hl3Mot0ltx2?auto=compress,format)


### 2. /devlog 커맨드 만들기 - "커맨드 하나로 워크플로우 정의"

세션 파일을 발견하고 나서,
`.claude/commands/devlog.md`
커맨드를 만들었어요.

세션 파일을 파싱해서 DEVLOG.md 형식으로 자동 정리해줘

놀라웠던 건,
`.md`
파일 하나로 복잡한 워크플로우를 정의할 수 있다는 거였어요. 코드 없이도 "세션 파일 읽고 → 파싱하고 → 포맷 맞춰서 → DEVLOG.md에 작성" 전체 프로세스를 명령할 수 있었습니다.

Claude가 커맨드 파일과 Python 예시 스크립트까지 작성해줬어요.

![보라색 배경의 한국어 앱 스크린샷](https://tribe-s3-production.imgix.net/D66v1kCCYW0LnRZXDOiAw?auto=compress,format)


### 3. /ai-case 커맨드 설계 - "DEVLOG → 사례글 파이프라인 완성"

DEVLOG를 만들고 나니, 이걸 기반으로 사례글까지 자동으로 쓰면 되겠다 싶었어요.

이 템플릿 기반으로 작성하기 전에 이 커맨드가 트리거되면

- 먼저 devlog파일을 읽고 파악한 다음

- 이 템플릿에 맞게 게시글을 쓰기 위해 너가 더 알아야 하는 맥락을 한 번에 하나씩 사용자에게 질문을 해줘

- 충분히 파악했다고 생각하면 md 파일로 작성해주고

- 최종 피드백 받아서 최종 수정하는 것까지 프로세스를 만들자

단순 템플릿이 아니라,
질문 → 맥락 파악 → 초안 → 피드백
전체 워크플로우를 커맨드로 정의했어요.

Claude가 8가지 핵심 질문 리스트까지 만들어줬습니다. 마치 인터뷰하듯이 글을 쓰는 구조가 됐죠.

![보라색 배경의 한국어 앱 스크린샷](https://tribe-s3-production.imgix.net/79kIbGEIFd3xw6r1mTL6r?auto=compress,format)


### 4. 스토리텔링 포맷으로 개선 - "너무 딱딱해!"

처음 만든 ai-case 템플릿이 "상황 - 요청 - 결과" 구조로 너무 딱딱했어요.

근데 너무 상황, 요청, 한일 이렇게 딱딱 구분되면 너무 읽기 어렵고 딱딱해. 스토리텔링으로 풀어야돼

Claude가 바로 자연스러운 서술체로 바꿔줬어요. 각 작업이 하나의 에피소드처럼 읽히도록 포맷을 변경해줬습니다.

결과:
읽는 사람 입장에서 훨씬 편하게 읽히는 글이 됐어요.

![](https://tribe-s3-production.imgix.net/LmzovYut2srvQvPWdazNK?auto=compress,format)


### 5. 막혔던 순간과 해결

문제 1: 포맷을 Claude가 마음대로 바꿔버림

DEVLOG 커맨드를 실행했더니 Claude가 기존 포맷을 무시하고 자기 스타일로 작성했어요.

해결:
"지금 포맷이 딱 좋거든?" 하고 직접적으로 피드백. 기존 DEVLOG.md를 예시로 보여주며 "이 포맷 그대로 유지해" 라고 명확히 지시했습니다.

문제 2: 사용자 질문을 너무 많이 빠뜨림

DEVLOG를 보니까 제가 했던 질문 중 많은 부분이 누락됐어요. "근데 일부만 되고 다 안채워지는데 이유가 뭐야?" 같은 중요한 후속 질문들이 빠졌죠.

해결:
커맨드에 "사용자 요청 정리 규칙" 섹션 추가. 짧은 질문도
맥락과 함께
기록하도록 명시했습니다.

예시:
`[Airtable 필드 자동 업데이트 중] 근데 일부만 되고 다 안채워지는데 이유가 뭐야?`

덧) 제목 추천 로직도 추가!

### 효과적이었던 것

- 솔직한 피드백이 최고
- "지금 포맷이 딱 좋거든?" 처럼 솔직하게 말하면 바로 고쳐준다

- 커맨드로 워크플로우 정의

- 실제로 써보고 불편한 점을 계속 피드백하며 다듬어야 진짜 쓸모있어짐

- 처음 만든 결과물에 만족

## 🌍 다른 업무에 적용한다면?

이 경험을 다른 상황에도 적용할 수 있을 것 같아요:

- 회의록 자동 생성
- 회의 녹음 → 텍스트 변환 → 커맨드로 요약 및 액션 아이템 추출

- 버그 리포트 자동 작성
- 에러 로그 → 커맨드로 재현 단계 및 해결 방법 정리

- 주간 업무 보고서 자동화
- 작업 기록 → 커맨드로 주간 요약 및 성과 정리

핵심은
"반복되는 문서 작업을 커맨드화"
하는 것이에요.
