# DevLog 질문 Flow 가이드

DevLog Writer는 **5개 핵심 질문**을 통해 사용자의 의도를 파악하고,
현재 세션의 맥락을 활용하여 자동으로 문서를 생성합니다.

---

## 질문 1️⃣: 작업 제목/주제

### 질문 내용
> 이번 세션에서 다룬 핵심 작업이 무엇인가요?

### 의도
- DevLog 파일명 생성 (`devlog_YYYYMMDD_{제목}.md`)
- 문서 제목 설정
- 메타정보의 "주제" 항목

### 작성 가이드
**형식:** 명사형 또는 동사형 모두 가능
**길이:** 5-10단어 권장
**스타일:** 간결하고 검색 가능한 키워드 포함

### 좋은 예시
- "DevLog 시스템 개선"
- "AI 리터러시 템플릿 개발"
- "다중 도구 세션 통합 설계"
- "Clarify 자동화 구조 재설계"
- "Think Partner Skill 마이그레이션"

### 피해야 할 예시
- "오늘 한 일" (너무 모호함)
- "버그 수정" (구체성 부족)
- "Claude Code를 활용한 프리랜서 자동화 워크플로우 개선 및 통합 시스템 구축" (너무 김)

### 활용
- **파일명:** `devlog_20260113_devlog-시스템-개선.md`
- **제목:** `# DevLog - 2026-01-13 (DevLog 시스템 개선)`
- **메타정보:** 주제 항목에 기록

## 질문 2️⃣: 문제 상황 (Before)

### 질문 내용
> 이 작업을 시작하게 된 배경이나 불편함이 무엇이었나요?

### 의도
- 작업의 동기와 맥락 파악
- 문제 인식 → 해결 과정의 스토리텔링 시작점
- "문제 상황 (Before)" 섹션 작성

### 작성 가이드
**형식:** 주관식, 자유로운 서술
**길이:** 2-5문장 권장
**스타일:** 구체적인 상황 묘사, 가능하면 정량적 데이터 포함

### 좋은 예시

**예시 1: 정량적 데이터 포함**
> 기존 DEVLOG.md는 단일 파일에 누적되어 200줄이 넘어가면서 특정 작업을 찾기 어려웠습니다.
> 세션 파일 파싱 후 수동으로 정리하는 데 매번 20-30분이 소요되었고, 형식도 일관되지 않았습니다.

**예시 2: 구체적 상황 묘사**
> 다중 도구(VSCode, AntiGravity 등)를 지원하려다 보니 자동화 복잡도가 높아졌습니다.
> 각 도구마다 세션 파일 형식이 달라서 파싱 로직을 별도로 작성해야 했고,
> 이를 통합하는 시스템 설계가 명확하지 않았습니다.

**예시 3: 반복 업무의 불편함**
> 매주 회고를 작성할 때 어디서부터 시작해야 할지 막막했습니다.
> 대화 내용을 다시 읽으며 핵심 작업을 추출하는 과정이 번거로웠고,
> 형식도 매번 바뀌어서 나중에 찾아보기 어려웠습니다.

### 피해야 할 예시
- "불편했어요" (너무 추상적)
- "기존 방식이 비효율적이었습니다" (구체성 부족)

### 활용
- **DevLog 본문:** "😫 문제 상황 (Before)" 섹션에 그대로 활용
- **블로그 변환 시:** 독자 공감 포인트 (스토리텔링 시작)
- **Before vs After 비교:** 개선 효과 강조

## 질문 3️⃣: Context의 역할

### 질문 내용
> 이번 작업에서 Claude Code 세션이 어떤 역할을 했나요?

### 의도
- AI 도구의 실질적 가치 인식
- 맥락 기반 작업의 이점 파악
- "🧠 Context의 역할" 섹션 작성

### 작성 가이드
**형식:** 불릿 포인트 또는 서술형
**길이:** 2-4개 항목 권장
**스타일:** 구체적인 작업 예시 + 효과 명시

### 좋은 예시

**예시 1: 수동 작업 감소**
- 현재 대화를 직접 활용해 수동 정리 시간을 20분 → 5분으로 단축
- 세션 파일 파싱 없이 Claude가 맥락을 이해하고 자동 문서화
- 질문에 답변만 하면 구조화된 DevLog가 자동 생성됨

**예시 2: 실시간 설계 협력**
- 복잡한 아키텍처를 실시간으로 논의하며 단계별 개선
- 막히는 부분을 즉시 질문하고 대안 제시받음
- 이전 대화 맥락을 유지하며 일관된 설계 가능

**예시 3: 학습과 기록 동시 진행**
- 작업 과정에서 배운 점을 Claude가 정리해줌
- 단순 명령 실행이 아니라 "왜 이렇게 했는지" 맥락 보존
- 나중에 다시 읽어도 이해 가능한 문서 생성

### 피해야 할 예시
- "Claude가 도움을 줬어요" (너무 추상적)
- "AI가 좋았습니다" (구체성 부족)

### 활용
- **DevLog 본문:** "🧠 Context의 역할" 섹션
- **AI 활용 인사이트:** 배운 점과 연계
- **블로그 변환 시:** AI 도구 활용 사례로 강조

## 질문 4️⃣: DevLog의 목적

### 질문 내용
> 이 DevLog를 어떤 용도로 활용할 예정인가요?

### 의도
- 사용자의 목표 파악
- 문서 톤앤매너 조정
- 메타정보 "목적" 항목 기록

### 선택 옵션

**A) 개인 회고용**
- 자신의 성장과 변화를 추적하는 개인 기록
- 매주/매월 회고 시 참고 자료
- 포트폴리오 작성 시 경험 요약

**B) 팀/커뮤니티 공유용**
- 팀원이나 같은 프리랜서들과 배운 점 공유
- 스터디 그룹, 커뮤니티에 지식 기여
- 협업 시 작업 맥락 전달

**C) 블로그/포스팅 기초 자료**
- DevLog → 블로그 변환 예정 (`/ai-case` 명령어 활용)
- 독자 친화적 스토리텔링 구조
- 타겟 독자 명확화 필요

**D) 포트폴리오 & 경력 자산**
- 보유 기술, 해결 경험, 성과 기록
- 이직/프로젝트 제안 시 활용
- 실무 경험 증빙 자료

**E) 기타 (주관식)**
- 사용자의 특수 목적
- 예: "향후 자동화 설계 시 레퍼런스", "팀 온보딩 자료" 등

### 작성 가이드
**형식:** 선택지 또는 주관식
**스타일:** 목적에 따라 문서 톤 조정

| 목적 | 문서 톤 | 강조점 |
|------|---------|--------|
| 개인 회고 | 솔직하고 성찰적 | 배운 점, 개선할 점 |
| 팀 공유 | 명확하고 전달 중심 | 작업 과정, 결정 이유 |
| 블로그 | 독자 친화적, 스토리텔링 | 문제 상황, 해결 과정 |
| 포트폴리오 | 성과 중심, 정량적 | 결과, 기술 스택 |

### 활용
- **메타정보:** "목적" 항목에 기록
- **문서 톤 조정:** 블로그용이면 더 친화적으로
- **후속 작업:** 블로그 변환 시 참고

## 질문 5️⃣: 주요 산출물

### 질문 내용
> 이번 작업의 주요 결과물이나 생성된 파일이 있다면?

### 의도
- 작업 결과 구체화
- 파일 경로 기록 (재사용, 참고용)
- "📌 DevLog 메타정보" 산출물 항목 작성

### 작성 가이드
**형식:** 파일 경로 리스트 또는 서술형
**스타일:** 구체적인 경로 + 간단한 설명

### 좋은 예시

**예시 1: 파일 경로 + 설명**
- `.claude/commands/devlog-writer.md` - DevLog 자동 생성 명령어
- `.claude/reference/devlog/template-guide.md` - 템플릿 구조 상세 가이드
- `.claude/reference/devlog/question-guide.md` - 질문 Flow 가이드
- `docs/_devlog/` - DevLog 저장 경로

**예시 2: 디렉토리 구조**
```
.claude/
  ├── commands/devlog-writer.md
  └── reference/devlog/
      ├── template-guide.md
      ├── question-guide.md
      └── examples.md
docs/
  └── _devlog/devlog_20260113_devlog-시스템-개선.md
```

**예시 3: 주요 변경사항**
- `clarify-automations/` 폴더 제거
- `think-partner.md` Skill → Command 마이그레이션
- `.claude/reference/clarify/` 신규 생성

### 피해야 할 예시
- "여러 파일 만들었어요" (구체성 부족)
- "파일 경로" (실제 경로 없음)

### 활용
- **메타정보:** "산출물" 항목에 기록
- **작업 과정 섹션:** 각 작업마다 생성 파일 명시
- **다음 단계:** 후속 작업 계획 수립 시 참고

## 🔄 질문 흐름 설계 의도

### 1. 제목 → 2. 문제 → 3. 도구 역할 → 4. 목적 → 5. 산출물

이 순서는 **스토리텔링 구조**를 따릅니다:

1. **무엇을** (제목)
2. **왜** (문제 상황)
3. **어떻게** (Context/도구 활용)
4. **누구를 위해** (목적/독자)
5. **무엇이 남았나** (산출물)

이 구조는:
- ✅ **자연스러운 대화 흐름**
- ✅ **블로그 변환에 최적화** (문제 → 해결 → 결과)
- ✅ **회고에 필요한 핵심 요소** 모두 포함

## 💡 질문 답변 팁

### 길이
- 각 질문당 2-5문장 권장
- 너무 짧으면 맥락 부족, 너무 길면 정리 부담

### 스타일
- **구체적으로:** "불편했어요" ❌ → "20분 걸렸어요" ✅
- **정량적으로:** "느렸어요" ❌ → "200줄 넘어갔어요" ✅
- **사실 중심:** "좋았어요" ❌ → "5분으로 단축됐어요" ✅

### 자유도
- 선택지(질문 4)는 힌트일 뿐, **주관식 답변 가능**
- 질문 의도를 벗어나도 괜찮음 (Claude가 해석)
- 답변 순서를 바꿔도 됨 (번호만 명시)

## 🎯 질문별 우선순위

**필수 질문:**
- 질문 1 (제목) - 파일명 생성에 필수
- 질문 2 (문제 상황) - DevLog 핵심 내용

**선택 질문:**
- 질문 3 (Context) - 생략 시 일반적 내용으로 대체
- 질문 4 (목적) - 생략 시 "개인 회고" 기본값
- 질문 5 (산출물) - 생략 시 Claude가 대화에서 추출

**즉, 시간이 부족하면 질문 1, 2만 답해도 됩니다.**
(향후 `/devlog-quick` 명령어는 이 방식으로 구현 예정)
