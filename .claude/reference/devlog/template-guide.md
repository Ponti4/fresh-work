## 📋 섹션별 작성 가이드

### 1. 한줄 요약
**목적:** 이 DevLog의 핵심을 한눈에 파악

**작성 팁:**
- "X를 Y로 개선했다"
- "A 문제를 B 방식으로 해결했다"
- 1-2문장으로 간결하게

**예시:**
> DevLog 시스템을 대화형 질문 방식으로 개선하여 수동 정리 과정을 5분으로 단축했다.

---

### 2. 이런 분들께 도움돼요
**목적:** 타겟 독자 명확화 (블로그 변환 시 활용)

**작성 팁:**
- 구체적인 상황/역할로 표현
- 3-5개 항목

**예시:**
- 프리랜서로 다양한 프로젝트를 관리하는 분
- AI 도구를 활용해 업무 자동화를 시도하는 비개발자
- 개발 경험을 체계적으로 기록하고 싶은 분

---

### 3. 문제 상황 (Before)
**목적:** 작업 시작 계기와 기존 불편함 명확화

**작성 팁:**
- 질문 2 답변을 그대로 활용
- 구체적인 상황 묘사
- 정량적 데이터 포함 시 더 좋음

**예시:**
> 기존 DEVLOG.md는 단일 파일에 누적되어 200줄이 넘어가면서 관리가 어려웠다.
> 세션 파일 파싱 후 수동으로 정리하는 데 매번 20-30분이 소요되었다.

---

### 4. Context의 역할
**목적:** Claude Code 세션의 가치 강조

**작성 팁:**
- 질문 3 답변 활용
- AI 도구의 실질적 이점 설명
- 맥락 인식의 효과 부각

**예시:**
- 현재 대화를 직접 활용해 수동 작업을 80% 감소
- 복잡한 아키텍처를 실시간으로 논의하며 단계별 개선
- 이전 작업 맥락을 유지하며 일관된 설계 가능

---

### 5. 작업 과정
**목적:** 실제 수행한 작업의 상세 기록

**작성 팁:**
- 현재 대화에서 주요 작업 추출
- 각 작업마다 "사용자 요청" + "Claude 작업" 구조
- 생성/수정된 파일 경로 포함

**예시:**
```markdown
### DevLog Writer 명령어 생성

**사용자 요청:**
```
대화형 질문 방식으로 DevLog를 자동 생성하는 명령어를 만들어줘.
```

**Claude 작업:**
- Interactive 질문 플로우 설계 (5개 핵심 질문)
- `.claude/commands/devlog-writer.md` 생성
- 템플릿 구조 정의 및 메타정보 포맷 확정
```

---

### 6. 결과 (After)
**목적:** 작업 전후 비교 및 성과 정량화

**작성 팁:**
- Before vs After 테이블로 명확한 비교
- 달성한 것을 체크리스트 형식으로
- 가능한 정량적 지표 포함

**예시:**
| 항목 | Before | After |
|------|--------|-------|
| DevLog 작성 시간 | 20-30분 (수동) | 5분 (대화형 질문) |
| 파일 구조 | 단일 파일 누적 | 날짜별 독립 파일 |
| 재사용성 | 낮음 | 블로그 변환 가능 |

---

### 7. 배운 점 & 인사이트
**목적:** 작업 과정에서 얻은 배움과 교훈

**작성 팁:**
- 기술적 인사이트
- AI 활용 패턴
- 다음 프로젝트에 적용할 수 있는 것

**예시:**
- Context 기반 대화는 수동 입력보다 자연스럽고 빠르다
- 질문 설계가 출력 품질을 결정한다 (GIGO 원칙)
- 날짜별 독립 파일 구조가 검색과 관리에 유리하다

---

### 8. 다음 단계
**목적:** 후속 작업 및 발전 방향 계획

**작성 팁:**
- 구체적인 액션 아이템
- 우선순위 표시 가능
- 다음 DevLog 연결 고리

**예시:**
- `/devlog-quick` 명령어 추가 (핵심 3개 질문만)
- 자동 태깅 시스템 도입
- 월간 DevLog 요약 대시보드 구축

---

### 9. DevLog 메타정보
**목적:** 문서 관리 및 검색을 위한 메타데이터

**작성 팁:**
- 작성일: YYYY-MM-DD 형식
- 주제: 질문 1 답변
- 목적: 질문 4 답변 (개인 회고/팀 공유/블로그/포트폴리오)
- 산출물: 질문 5 답변 (파일 경로 리스트)

---

## 🎯 작성 시 주의사항

1. **맥락 활용:** 현재 대화의 실제 내용을 적극 활용
2. **구체성:** 추상적 표현보다 구체적 예시 선호
3. **정량화:** 가능한 숫자로 표현 (시간, 라인 수, 파일 개수 등)
4. **재사용성:** 블로그 변환을 고려한 구조 유지
5. **독립성:** 각 DevLog는 독립적으로 이해 가능해야 함

---

## 🔄 블로그 변환 대비

DevLog는 개인 회고용이지만, 필요시 블로그 포스팅으로 변환될 수 있습니다.

**변환 시 활용되는 섹션:**
- 한줄 요약 → 블로그 서론
- 타겟 독자 → 블로그 대상 명확화
- 문제 상황 → 스토리텔링 시작점
- 작업 과정 → 본문 구조
- 배운 점 → 핵심 메시지
- 다음 단계 → 독자 행동 유도 (CTA)

**즉, 구조적으로 작성하면 블로그 변환이 용이합니다.**
